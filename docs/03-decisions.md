# 关键决策与思路：为什么这么做

> 本篇说明：项目里几个重要的设计选择，每条给"决策 + 理由"。
> 如果你要改架构或扩展功能，先看这里，能避免踩已经踩过的坑。

---

## 决策 1：做成 Skill，不做 MCP

**决策**：所有工具做成 Claude Code skill（`skills/<name>/SKILL.md` + 可选脚本），而不是 MCP（Model Context Protocol 服务器）。

**理由**：
- MCP 需要单独搭建服务器进程并维护，只有需要"集中管理复杂逻辑 / 长期存储登录状态 / 多个客户端共享同一个服务"时才划算。
- 我们的需求是"聊天里随口查"——交互式、一次性调研，不是需要常驻服务的场景。
- Reddit 主路径纯靠 AI 的 WebFetch 抓 Arctic Shift API；YouTube 纯靠 WebFetch 抓官方接口；X 有现成的 twikit 运行环境在服务器上。这些都不需要 MCP。
- Skill 跨 AI 成员共享有现成机制（`sync-codex-skills.sh`），部署成本低。

---

## 决策 2：发布走"每平台原生"，不用聚合工具

**决策**：Reddit / YouTube / X 的发布各自用原生方式实现（PRAW / YouTube Data API / twikit），而不是接入 Postiz / Buffer 等聚合工具。

**理由**：
- 聚合工具**不支持发评论**（只支持发帖），而我们的主要用途是回复评论、参与社区讨论，不是广播式发帖。
- YouTube 写操作已有官方 API，Reddit 有 PRAW，聚合工具多一层中间商没必要。
- X 必须走服务器 twikit，聚合工具也复用不了那套环境。
- 原生方式能更精细地控制每个平台的行为和参数。

---

## 决策 3：X 必须走海外服务器

**决策**：X 的搜索和发推不在本机跑，通过 SSH 发到硅谷服务器（47.251.0.142）执行。

**理由**：
- **双重障碍**：本机国内 IP 过不了 X 反爬（Arkose 挑战页 / 影子屏蔽），加上 twikit 本身有 `KEY_BYTE indices` 的反爬 bug（详见踩坑台账）。
- 服务器能成的三个条件：① 美国阿里云 IP（X 认） ② twikit 已打补丁（绕过 transaction-id 反爬） ③ 有效 cookie（账号登录态）。
- 本机只需要 `paramiko`（SSH 客户端），不需要 twikit。把搜索/发推任务发到服务器，取回 JSON。

## 决策 4：服务器工具箱独立（不复用 ai-infohub）

**决策**：新建 `C:\QuriovXTools\` 作为独立工具箱，不把 x-research / x-publish 塞进已有的 `C:\AIInfoHub\`。

**理由**：
- `AIInfoHub` 跑着每日定时日报管线（关注账号推文 → 去重打分 → 翻译 → 飞书），是生产系统，不能随便动。
- 两个系统用途不同（日报管线 vs 交互式搜索/发推），cookie 角色也不同（固定账号 vs 可配搜索/发布双账号）。
- 独立工具箱让我们能自由更新补丁、换 cookie，不影响日报管线稳定性。

---

## 决策 5：默认 dry-run 安全门

**决策**：所有三个"发"的 skill（x-publish / youtube-publish / reddit-publish）都设计成**默认 dry-run**——不加 `--send` 参数就只验证 + 预览，绝不真发。

**理由**：
- 发布操作不可逆（发出去了就是发出去了），错发代价大。
- 社媒账号（尤其品牌号）一旦被判垃圾、被封，损失难以挽回。
- AI 代理执行时更容易出意外——强制要求"加 `--send` 才真发"是最低成本的安全网。
- 干净的体验：每次先看预览（dry-run 输出将发的内容 + 模式），确认无误再加 `--send`。

---

## 决策 6：凭据走 `.secrets/` 目录，不入库

**决策**：所有真实凭据（服务器密码、cookie、OAuth token、API key）存在各 skill 的 `.secrets/` 目录里，`.gitignore` 隔离，绝不入 GitHub。

**理由**：
- 凭据一旦进 git 就等于公开，账号被盗几乎不可避免。
- 每个 skill 都提供 `*.json.example` 模板，接手者照模板创建、填入真实值即可。
- 团队多人用同一个仓库，但每人管理自己本地的凭据（找负责人要），不通过仓库传递。
- 这是业界标准做法，没有理由打破。

---

## 决策 7：YouTube 发评论只做"评论"，不做"上传视频"

**决策**：youtube-publish skill 只实现发评论和回复评论，不实现上传视频。

**理由**：
- 上传视频需要 OAuth 写入范围 + 视频文件，成本高、用途特殊，目前没有需求。
- 发评论是"参与社区讨论"的核心用途，和 x-research / reddit-research 调研到的内容形成闭环（研究用户怎么说 → 去回应他们）。
- 视频上传是另一个独立的功能，如果之后需要再另建 skill。

---

## 决策 8："内容来源自动生成"那层暂缓

**决策**：工具先做"给它内容就发"，自动生成发布内容的那一层暂缓，等用户想清楚再加。

**理由**：
- 内容生成 + 发布是两个独立的问题，混在一起会让每个部分都更复杂、更难验证。
- 先验证"发"这一层能稳定工作，再考虑"自动生成什么内容发"。
- 用户对"应该发什么"还没有明确策略，贸然自动化可能生成错误的内容。
