# AI 协作开发手册

> 基于 E-Commerce Database 项目实战经验提炼，适用于 AI + 人类协作的软件开发场景。

---

## 体系一：AI 闭环测试

### 核心原则
**"功能存在" ≠ "功能可用"**。每个功能必须通过真实交互测试（点击/输入/提交），而不仅仅是 DOM 元素存在检查。

### 测试金字塔
```
Layer 4: 端到端流程测试 (最重要，模拟完整业务流程)
Layer 3: 视觉回归 (截图对比 + getComputedStyle)
Layer 2: API 合约测试 (结构、字段、计算结果)
Layer 1: 单元测试 (计算逻辑)
```

### 实施规则

1. **每次代码改动后必须运行测试**
   - 改了哪个页面就跑哪个页面的测试: `npm run test:page:<name>`
   - 报告完成前跑全套: `npm run test:check`

2. **新功能必须有交互测试**
   - 不是检查 DOM 存在，而是模拟用户操作（click → type → submit → verify result）
   - 检查 happy path + error path

3. **视觉样式用 getComputedStyle 验证**
   - 不用 `element.style`（CSS !important 会覆盖 inline style）
   - Handsontable 的 htDimmed class 是血泪教训

4. **端到端流程测试**
   - 在关键里程碑后必须跑一遍完整业务流程
   - 例: 核价 → 审批 → 创建商品 → 创建SKU → 商品管理验证

### 自动化执行
- `dev-loop-check.js` Stop hook: 自动检测"完成"声明是否有测试证据
- `SCOPE-CHECK`: 自动检测用户列出多个需求时是否全部实现

## 体系二：深层意图分析

### 核心原则
用户说的是 Layer 1（表面需求），你需要找到 Layer 3（深层目标），并在 Layer 3 层面提出更好的解决方案。

### 三层分析框架
```
Layer 1: 表面需求 — 用户字面上要求的（"加一个列宽调整功能"）
Layer 2: 直接目的 — 为什么要这个（"有些列太宽，重要信息看不到"）
Layer 3: 深层目标 — 真正在优化什么（"运营需要一眼看到所有关键信息"）
```

### 执行时机
- **收到任何功能请求或优化建议时**，必须先做三层分析
- 分析完后主动提出 Layer 3 层面的替代方案
- 如果 Layer 3 揭示了更好的路径，直接建议

### 典型案例
| 用户说 | Layer 3 | 更好的方案 |
|--------|---------|-----------|
| "加列宽拖拽" | 运营要一屏看到关键信息 | 拖拽 + 智能默认宽度 + 布局持久化 |
| "每列允许拖动排序" | 自定义工作区 | 字段配置面板（飞书风格）+ 视图保存 |
| "批量选择4x4单元格" | 高效数据迁移 | 智能粘贴 + 导出功能 |

### 何时不需要
- Bug 修复（明确的重现步骤）
- 技术指令（"用 antd Table 不用 ProTable"）
- 简单事实问题

## 体系三：复盘迭代优化

### 核心原则
每次多步骤任务完成后，检查是否有浪费的纠正轮次。如果有，提取可复用的教训并固化。

### 触发条件
- 任务花了 3+ 轮纠正
- 多阶段功能完成后
- 用户报告了遗漏
- 测试通过但功能实际不可用（假阳性）

### 复盘格式
```
### [问题标题]
- **症状**: 用户看到了什么 / 什么失败了
- **根因**: 为什么发生（不是表面原因）
- **浪费轮次**: 多少个额外循环
- **预防**: 添加了什么规则/工具/测试
- **范围**: 仅此项目 / 跨项目通用
```

### 教训固化位置
| 教训类型 | 存储位置 | 例子 |
|---------|---------|------|
| 行为规则 | ~/.claude/rules/common/ | "必须用 getComputedStyle" |
| 项目模式 | <project>/.claude/playbook.md | "antd Table 有 aria-hidden 行" |
| 测试辅助 | tests/e2e/helpers/ | 共享选择器 |
| 用户偏好 | memory/feedback_*.md | "不要尾部总结" |

### 关键反模式（血泪教训）
1. **先报完成再测试**: 声称"已修复"但没跑测试
2. **表面检查**: 检查 DOM 属性而非 computed/rendered 状态
3. **Todo 替换丢失**: 完成一些任务后新建 todo list，丢掉了旧的 pending 项
4. **范围遗忘**: 用户列了 7 个需求，实现了 4 个就报"完成"
5. **功能存在≠功能可用**: 按钮存在但点击报错，窗口存在但 reload 整页

### 执行保障
- `dev-loop-check.js` hook 自动检测完成声明和纠正轮次
- `ai-dev-loop.md` 规则每次对话自动加载
- `playbook.md` 每次复盘后更新

## 快速参考卡

```
收到需求 → 意图分析(3层) → 规划 → 实施 → 测试(交互级) → 逐条检查 → 报告
                                                    ↓
                                              失败? → 修复 → 重测
                                                    ↓
                                              完成 → 复盘 → 固化教训
```
