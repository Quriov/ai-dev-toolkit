# /retro — 复盘迭代优化

当完成一个多步骤任务、用户指出遗漏、或 dev-loop-check hook 触发 [RETRO] 警告时，执行此复盘流程。

## 执行步骤

### Step 1: 识别浪费的纠正轮次
回顾当前对话，找出：
- 用户说了"没解决"、"还是不行"、"这个没做"等纠正信号的地方
- 同一个功能改了 2 次以上的地方
- 声称"完成"但实际有问题的地方

### Step 2: 对每个问题填写复盘模板
```
### [问题标题]
- **症状**: 用户看到了什么 / 什么失败了
- **根因**: 为什么发生（5 Whys，找到根本原因而非表面原因）
- **浪费轮次**: 多少个额外循环
- **预防**: 需要添加什么规则/工具/测试来防止复发
- **范围**: 仅此项目(playbook) / 跨项目通用(rules)
```

### Step 3: 固化教训
根据范围，将教训写入对应位置：

| 范围 | 位置 | 操作 |
|------|------|------|
| 跨项目行为规则 | `~/.claude/rules/common/ai-dev-loop.md` | 添加到 Anti-Patterns 或 Checklist |
| 项目特有模式 | `<project>/.claude/playbook.md` | 添加到 Revision Log + 对应章节 |
| 测试辅助 | `tests/e2e/helpers/` | 创建/更新共享测试工具 |
| 用户偏好 | `memory/feedback_*.md` | 创建/更新偏好记忆 |

### Step 4: 更新 dev-loop-check hook（如果发现新的检测模式）
如果复盘发现了一种可以程序化检测的问题模式，更新 `~/.claude/hooks/dev-loop-check.js` 添加自动检测。

### Step 5: 输出复盘报告
向用户输出简洁的复盘总结：
- 发现了 N 个问题
- 固化了 M 条教训
- 位置：列出修改的文件

## 何时不需要复盘
- 任务一次通过，没有纠正轮次
- 纯信息查询/讨论（没有代码改动）
- 用户明确说"不需要复盘"
